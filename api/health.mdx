---
title: 'Health Check'
api: 'GET /api/health'
description: 'Check the backend API server status and configuration'
---

Returns the health status and basic configuration information of the backend API server. This endpoint is useful for monitoring, health checks, and verifying the server configuration.

## Authentication

This endpoint does not require authentication and is publicly accessible.

## Response

<ResponseField name="status" type="string" required>
  Current health status of the server. Always returns `"ok"` if the server is running.
</ResponseField>

<ResponseField name="simulationMode" type="boolean" required>
  Indicates whether the server is running in simulation mode.
  
  - `true`: Simulation mode is enabled (using mock data)
  - `false`: Production mode (connecting to real RD Connection Broker)
</ResponseField>

<ResponseField name="rdcbServer" type="string" required>
  The hostname or IP address of the configured RD Connection Broker server.
</ResponseField>

<ResponseField name="timestamp" type="string" required>
  ISO 8601 timestamp of when the health check was performed.
</ResponseField>

## Use Cases

### Monitoring and Alerting

Use this endpoint to monitor your RDSWeb Custom deployment:

```bash
#!/bin/bash
# Simple health check script

HEALTH_URL="https://your-server.com/api/health"
RESPONSE=$(curl -s $HEALTH_URL)
STATUS=$(echo $RESPONSE | jq -r '.status')

if [ "$STATUS" != "ok" ]; then
  echo "Health check failed!"
  # Send alert notification
  exit 1
fi

echo "Server is healthy"
```

### Kubernetes/Docker Health Probes

Configure container health checks:

```yaml
livenessProbe:
  httpGet:
    path: /api/health
    port: 3000
  initialDelaySeconds: 10
  periodSeconds: 30

readinessProbe:
  httpGet:
    path: /api/health
    port: 3000
  initialDelaySeconds: 5
  periodSeconds: 10
```

### Load Balancer Health Checks

Configure your load balancer to use this endpoint for health monitoring:

- **Path**: `/api/health`
- **Expected Status Code**: `200`
- **Expected Response**: `{"status":"ok"}`
- **Timeout**: 5 seconds
- **Interval**: 30 seconds

## Configuration Detection

The `simulationMode` and `rdcbServer` fields help you verify the server configuration:

<Tabs>
  <Tab title="Development">
    **Typical development configuration**
    
    ```json
    {
      "status": "ok",
      "simulationMode": true,
      "rdcbServer": "localhost",
      "timestamp": "2026-03-04T15:30:45.123Z"
    }
    ```
    
    Simulation mode is enabled, allowing development and testing without requiring RDS infrastructure.
  </Tab>
  
  <Tab title="Production">
    **Typical production configuration**
    
    ```json
    {
      "status": "ok",
      "simulationMode": false,
      "rdcbServer": "rdcb.yourdomain.local",
      "timestamp": "2026-03-04T15:30:45.123Z"
    }
    ```
    
    Production mode connects to a real RD Connection Broker server.
  </Tab>
</Tabs>

<Info>
  The health check endpoint always returns status code `200` if the server is running, even if it cannot connect to the RD Connection Broker. For more detailed connectivity checks, use the `/api/apps` endpoint which actually queries the RDCB.
</Info>

<Warning>
  This endpoint exposes the RDCB server hostname. In security-sensitive environments, consider restricting access to this endpoint or removing the `rdcbServer` field from the response.
</Warning>

## Response Time

The health check endpoint is designed to respond quickly:

- **Typical response time**: < 10ms
- **No external dependencies**: Does not query the RD Connection Broker or Active Directory
- **No database queries**: Returns configuration values from memory

<Tip>
  For a comprehensive health check that validates connectivity to the RD Connection Broker, make an authenticated request to `/api/apps` and verify it returns a successful response.
</Tip>
